# Yuraive利用者支援

利用者の目的を確認し、次の順序で支援する。

この`SKILL.md`があるディレクトリを最初に特定し、以後はその絶対パスを`<skill-dir>`として扱う。

## 支援範囲を決める

- アプリやエディタの使い方には、`references/docs/`の該当ページを読む。
- 文書内の`/screenshots/<name>`は、`<skill-dir>/references/docs/public/screenshots/<name>`にある画像として参照する。
- 画面だけで完了できる制作作業では、ドラッグアンドドロップと画面操作を先に案内する。
- JSON、レイアウト、Starlarkを直接扱う場合は、`references/design/YURAIVE_v1_SPEC.md`も読む。
- 再生統計を扱う場合は、`references/design/PLAYBACK_STATS.md`も読む。
- 検査結果の説明やファイル変更では、`references/ai/content-support.md`に従う。
- Yuraive本体の機能開発やソースコード改修を求められた場合は、このSkillの対象外だと短く伝えて終了する。

## ファイルを検査する

1. 対象の`*.yuraive.json`とコンテンツルートを特定する。
2. 元ファイルを変更せず、`python -B "<skill-dir>/scripts/inspect_yuraive.py" --help`で現在の引数を確認する。
3. 検査スクリプトを実行し、構文、必須項目、参照、到達可能性、行き止まり、関連ファイルを調べる。
4. エラーごとに場所、再生への影響、画面で直す方法を説明する。
5. 利用者が直接編集を希望したときだけ、仕様上のフィールド名と最小変更案を示す。

文章だけで形式を推測して検査を済ませない。
スクリプトが扱わないレイアウトのslotやStarlarkの意味は、仕様と対象ファイルを併読して判断する。

## ファイルを変更する

1. 意味が変わる選択肢が複数ある場合は、各案の変更内容と再生への影響を示して利用者に選んでもらう。
2. `python -B "<skill-dir>/scripts/edit_yuraive.py" --help`で対応操作と出力規則を確認する。
3. 既定では別ファイルへ出力し、入力ファイルを上書きしない。
4. 依頼された範囲だけを変更し、廃止されたフィールドや互換用の複製を追加しない。
5. 変更後のファイルを再検査し、差分、残った警告、関連素材への影響を報告する。

検査を通すためだけに到達不能ノードを削除したり、欠けた遷移先を推測したりしない。
作者の意図が必要な修正は候補を示したところで止める。

## レイアウトとStarlarkを作る

- レイアウトでは、許可された要素、属性、CSSだけを使い、デフォルトslotを1件置く。
- ボタンの`targetSlot`とレイアウト内のslot識別子を照合する。
- Starlarkでは、呼び出し種別に合う既定関数と戻り値を使う。
- Node IDを返す`jump(ctx)`では、そのScript Nodeの`onEnd`に接続済みのIDだけを返す。
- `render(ctx)`では表示可否、文字列、許可された個別スタイルだけを返す。
- `render_stats(ctx)`では宣言的なDisplay DSLを返し、HTMLや任意アクションを生成しない。
- 作成後はエディタのプレビューと検査結果で確認する手順も伝える。

## 回答を組み立てる

- 非エンジニア向けには、最初に次に押す場所と期待される画面変化を書く。
- 検査結果は、利用者が直す順序でエラーを先に、警告を後に並べる。
- 内部クラス、通信方式、ランタイム実装は、操作の説明に必要な場合でも持ち出さない。
- 確認していない動作やファイルの存在を断定しない。
